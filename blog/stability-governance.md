# 千万级 DAU 应用的稳定性治理体系建设（从 0 到 1 到规模化）

## 一、背景与问题定义

### 1.1 我们面对的是什么场景

滴滴货运司机端是一个千万级 DAU 的移动应用，运行在 Android、 iOS 和 HarmonyOS 三端之上，承载着接单、导航、沟通、支付等核心货运链路。司机端的特点是：

- **使用频次高、单次时长长**：司机在出工期间几乎全程保持应用前台运行
- **稳定性直接影响收入**：一次崩溃可能导致司机错过订单、导航中断、甚至无法完成运输
- **场景复杂**：涉及 GPS 定位、音视频通话、地图渲染、蓝牙外设等多种重资源操作
- **多端碎片化**：Android 机型碎片化严重，iOS 版本迭代快，HarmonyOS 生态尚在演进中

当应用规模达到千万级 DAU 时，一个千分之一的崩溃率就意味着每天上万次崩溃。对于司机端而言，每一次崩溃都有真实的业务损失。

### 1.2 为什么"单点优化"解决不了问题

在早期阶段，我们的稳定性治理是典型的"救火模式"：

> 线上出了问题 → 定位修复 → 发版 → 下次再出另一个问题

这个模式有几个结构性缺陷：

**第一，被动响应而非主动防御。** 问题已经影响了用户，我们才去修复。对于货运业务，这意味着实打实的订单损失和司机投诉。

**第二，单点修复无法防止同类问题。** 修复了一个 NullPointerException，下一个 IllegalStateException 还会出现。问题的根因是缺乏系统性的预防机制，而不是某个具体 bug。

**第三，缺乏可观测性导致无法评估全貌。** 我们知道自己修复了什么，但不知道"还有多少问题没被发现"。没有量化就没有治理。

**第四，团队协作成本随规模非线性增长。** 当多人多地协作开发时，一个人的疏忽可能导致全量用户的稳定性受损。纯靠"人肉 Review"和"责任心"是不可规模化的。

> 这类问题本质上是一个**复杂系统的可信度工程问题**：当系统规模、团队规模、业务复杂度同时增长时，如何用工程化手段保证系统的确定性输出？

## 二、稳定性治理的核心原则

在搭建体系之前，我们首先明确了几个指导原则。这些原则不是事后总结的漂亮话，而是在方案选型和资源投入时实际用来做决策的判断依据。

### 原则一：分层治理，每层设防

稳定性问题可以在不同层面被拦截：代码层、构建层、测试层、灰度层、线上层。每一层都应该有对应的防护手段，且下层不应依赖上层发现问题。理想状态下，一个问题应该尽可能在更早的层级被发现。

### 原则二：可观测性优先于稳定性本身

你无法治理你观测不到的东西。在很多情况下，"先让问题可见"比"立刻修复问题"更重要。一个可见的问题可以被定位、被复现、被归因，最终被系统性解决。一个不可见的问题永远只会被当作"偶发"。

### 原则三：核心链路优先

不是所有功能都值得同等的稳定性投入。接单链路、订单流转、支付是核心链路，它们的稳定性直接影响业务收入。而一些配置页、反馈页等非核心功能，在资源有限时可以接受相对较低的治理标准。

### 原则四：体系化而非点式优化

单点修复只能解决"这一次"的问题。体系化的目标是让"这一类"问题不再发生。每一次事故复盘，不仅要找到根因，还要反问：我们的流程、工具、机制中缺少了什么，才让这个问题流到了线上？

### 原则五：用机制代替依赖

不依赖人的责任心，而是依赖机制和工具来保证质量。人都会犯错，但好的机制可以把错误的伤害降到最低。

## 三、整体架构设计

基于上述原则，我们构建了一个"事前-事中-事后"三阶段闭环的稳定性治理体系。

### 3.1 体系全景

```
┌─────────────────────────────────────────────────────────┐
│                    稳定性治理体系                          │
├──────────────┬──────────────┬────────────────────────────┤
│   事前（预防）  │   事中（监控）  │    事后（复盘与改进）        │
├──────────────┼──────────────┼────────────────────────────┤
│ 发版 Checklist │ Crash 实时监控  │ 事故自动归因               │
│ CR 机制       │ 卡顿监控        │ TODO 跟踪与闭环            │
│ 稳定分机制     │ 白屏监控        │ 知识库沉淀                 │
│ 自动化测试     │ JS异常监控      │ 流程改进                   │
│ 灰度放量策略   │ 灰度维度聚合    │                          │
│ Bug Hunter    │ 分业务场景监控   │                          │
│ 集成回归       │ 告警体系        │                          │
└──────────────┴──────────────┴────────────────────────────┘
```

### 3.2 事前（预防层）

前移稳定性左盾，在问题产生之前就将其拦截。核心手段：

- **发版 Checklist**：标准化发版前的检查项，从代码合入到灰度放量的每个环节设置卡口
- **CR 机制**：强制 Code Review，并且在 CR 中加入稳定性相关的检查要点
- **稳定分机制**：对每个模块/业务线进行稳定性量化评分，低于阈值的模块不允许发版
- **灰度放量策略**：白名单 → 分城市 → 分阶段放量，每阶段有明确的观测窗口
- **Bug Hunter**：通过有奖活动激励全员在灰度阶段主动发现并上报问题

### 3.3 事中（监控层）

核心目标是：**让每一个线上问题都在 5 分钟内被发现，且具备可追溯信息。**

监控覆盖五个维度：

1. **Crash 监控**：实时上报、堆栈聚合、版本对比、影响面评估
2. **卡顿监控**：帧率采集、主线程阻塞检测、ANR 归因
3. **白屏监控**：自研 SDK，页面级白屏识别（后面详述）
4. **JS 异常监控**：try-catch 兜底、React Component Tree 异常边界、未捕获 Promise 异常
5. **灰度监控**：按城市、版本、机型维度聚合，与基线版本自动对比

### 3.4 事后（复盘层）

- **事故自动归因**：结合监控数据和变更记录，缩小问题范围
- **TODO 跟踪与闭环**：每个事故生成改进事项，跟踪到具体负责人和截止日期
- **知识库沉淀**：每类问题的根因、修复方案、检查要点沉淀为文档，反哺事前预防

## 四、关键设计与实践

### 4.1 Checklist 机制：如何让"检查"真正起作用

Checklist 是事前预防中最简单也最容易流于形式的工具。如果我们只是列一张表让开发逐项打勾，最终只会变成"为了发版而打勾"的走过场。

我们的做法是：

**Checklist 与灰度流程绑定。** 不完成 Checklist 无法进入灰度放量环节。这不是技术限制，而是流程卡口——工具提供状态标记，流程要求状态通过。

**Checklist 持续演进。** 每次事故复盘中发现的新问题类型，都会抽象成一条新的检查项加入 Checklist。它不是一个静态文档，而是一个由事后经验反哺的动态工具。

**Checklist 分层分级。** 核心业务变更需要更严格的 Checklist，工具型/配置型变更可以简化。不同等级的变更走不同的检查路径。

### 4.2 灰度监控设计：如何在不影响全量的前提下发现问题

灰度放量的本质是**一个受控的实验**。我们需要在灰度期间做出判断：新版本是否比基线版本更差。

关键设计要点：

**对比基线而非绝对数值。** 某个版本的 Crash 率是 0.5%，看起来不错，但如果基线版本是 0.1%，这 0.5% 就是一个需要回滚的信号。灰度监控的核心逻辑是和同时间段的基线版本做同比对比。

**分维度聚合。** 同一版本在不同城市、不同机型上的表现可能完全不同。按业务场景、机型、系统版本、城市等维度分别计算指标，避免均值掩盖问题。

**设定告警阈值和自动回滚条件。** 当核心指标劣化超过阈值时，自动触发告警。对于 P0 级指标劣化，支持一键回滚。

### 4.3 白屏治理方案：从不可见到可监控

白屏是一个特殊且棘手的问题：它不像 Crash 那样有明显的异常堆栈，用户看到的是白屏，但应用进程还在正常运行，所以传统的 Crash 监控完全覆盖不到。

**我们面临的核心问题：** 如何定义白屏？如何区分"正在加载中"和"真的白屏了"？如何在不侵入业务代码的前提下实现监控？

**我们的方案：**

自研白屏监控 SDK，核心思路是"视觉验证 + 兜底超时"的双重判定：

1. **首屏元素检测**：页面加载完成后，检测关键 DOM 元素或 Native View 是否渲染成功。如果超过超时时间仍未检测到预期元素，判定为白屏。
2. **灰度聚合**：白屏事件本身可能有误报（网络慢导致加载超时），但按灰度维度聚合后，如果某个版本的某类页面白屏率显著高于基线，就是明确的告警信号。

这个方案的价值在于：它让白屏从一个"只存在于用户投诉中的问题"变成了一个"有数据、有趋势、可比较、可预警"的指标。

> 这类问题本质上是一个**感知与事实的鸿沟问题**：用户感知到的很多"不好用"并未被系统的监控体系捕获，而我们的工作就是不断缩小这个鸿沟。

实际运行中，这套 SDK 成功召回了以下类型的问题：

- **域名异常**：第三方 CDN 域名解析失败导致页面空白
- **H5 图片加载慢**：图片资源没有走 CDN 或 CDN 预热不到位，导致超时白屏
- **页面配置错误**：后端返回的页面配置数据异常，导致前端渲染中断但不报错

### 4.4 JS 异常治理：从千万级到可监控可召回

JS 异常在 Hybrid 架构中是一个特别棘手的问题：异常发生频率极高（千万级/天），但大部分是"无害异常"（比如第三方 SDK 的异常被 try-catch 兜住了，或者是在非核心路径上）。

**如果不加区分地全量上报，监控系统会直接被淹没，真正的有效告警反而被噪音覆盖。**

我们的治理策略分三步走：

**第一步：异常分级。** 将 JS 异常按严重程度分为三类：
- **Fatal**：影响核心功能，导致页面不可交互
- **Warning**：功能降级但不影响主流程
- **Info**：被 try-catch 捕获的已知异常

**第二步：噪音过滤。** 对已知的、无害的异常建立白名单，不产生告警。同时通过堆栈聚合，将同一类异常合并统计。

**第三步：趋势监控。** 不关注单次异常，而是关注某个版本的异常率相比基线是否有显著上升。趋势异常往往预示着新引入的代码缺陷。

治理结果：JS 异常从"千万级的噪音海洋"变成了"可监控、可召回、可定位"的质量指标。

### 4.5 如何保证"问题可召回"

可召回意味着：当线上出了问题，我们能在多长时间内知道、定位到根因、并且有足够的信息去修复。

召回能力的核心是**链路追踪和现场保留**。具体做法：

- **关键路径埋点**：在核心业务链路上埋入 Trace 信息，异常时自动携带上下文
- **堆栈聚合**：同一堆栈的异常自动归并，便于按影响面排序
- **用户反馈通道**：简化司机上报问题的流程，引导用户在遇到异常时主动上报，补充监控盲区

## 五、难点与取舍

### 5.1 为什么不全量监控

这是一个经常被问到的问题：为什么我们不把所有指标都监控起来？

答案很简单：**监控本身也有成本。** 全量监控意味着全量日志、全量上报、全量存储，这对客户端性能和后端带宽都是不可忽视的负担。具体来说：

- **客户端性能开销**：每条日志的采集、序列化、写入都消耗 CPU 和 IO
- **网络带宽**：移动端上传日志消耗用户流量，也消耗服务端带宽
- **存储与计算成本**：千万级 DAU 的全量日志，存储成本是巨大的

> 这类问题本质上是一个**信息价值与成本的权衡问题**：每多采集一条信息，都有边际成本，但边际价值在递减。

我们的策略是"分层采样"：

- Crash 类问题全量上报（因为频率低、价值高）
- JS 异常部分采样（频率高、大部分无害）
- 性能数据按比例采样（统计性信息不需要全量）
- 白屏事件灰度聚合后上报（不上报原始事件，只上报聚合后的指标）

### 5.2 如何平衡性能 vs 日志

稳定性监控和性能本身就是一对矛盾。监控越详细，对应用性能的影响越大。

我们的决策原则是：

- **异步操作**：所有日志采集和上报都在后台线程进行，不阻塞主线程
- **批量上传**：日志先写入本地缓冲区，达到一定量后批量上传，减少网络请求次数
- **上传时机优化**：只在 WiFi 环境和应用后台时上传非紧急日志，避免影响用户正在使用的体验
- **可配置的上报级别**：线上版本使用轻量级监控，内测版本使用全量详细日志

### 5.3 如何避免监控噪音

监控噪音是稳定性治理中最大的敌人之一。过多的无效告警会导致"狼来了"效应——开发团队对告警麻木，真正的有效告警反而被忽略。

噪音治理的几个关键做法：

- **聚合而非逐条告警**：不告警单次异常，而是告警异常率的显著变化
- **动态基线**：基线不是固定的，而是根据历史数据动态计算的。节假日、大促期间的正常波动不应触发告警
- **静默期机制**：新版本发布后，允许一定时间的静默期，让系统自动收集基线数据后再开启异常告警
- **分级告警**：P0（立即响应）、P1（30分钟内响应）、P2（当天内响应）、P3（记录不告警）

### 5.4 如何推动团队执行

这可能是整个稳定性治理体系中最难的部分。工具和流程可以设计得很完善，但如果团队不执行，一切都是空谈。

**第一，规则要足够硬。** 发版 Checklist 不完成 = 不能发版。这不是建议，是硬性要求。技术上我们要在 CI/CD 流程中设置卡口。

**第二，反馈要足够快。** 每次灰度放量后，稳定性数据要在 1 小时内同步给相关开发。好和坏都要让团队尽快知道。

**第三，透明化。** 建立稳定性看板，每个模块的 Crash 率、卡顿率、JS 异常率对所有团队可见。没有什么比"你的模块稳定性排名垫底"更能推动改进了。

**第四，正向激励。** Bug Hunter 活动、稳定性之星评选、优秀复盘案例分享——让稳定性治理从"被迫做"变成"愿意做"。

## 六、结果与收益

### 6.1 核心指标变化

经过持续治理，司机端关键稳定性指标实现了数量级级别的改善：

| 指标 | 治理前 | 治理后 | 改善幅度 |
|------|--------|--------|----------|
| Crash 率 | 2% | 0.4% | 5x 改善 |
| 卡顿率 | 3% | 0.9% | 3.3x 改善 |
| JS 异常 | 千万级/天（噪音） | 可监控可召回 | 定性改善 |
| Checklist 拦截问题 | 无 | 司机端 20+ / 货主端 200+ | 从 0 到 1 |
| 主动上报问题 | 无体系 | 29 个 | 从 0 到 1 |

### 6.2 团队层面的变化

比数据更重要的是团队质量意识的转变。具体体现为：

- **从"Crash 率达标就行"到"任何一个指标异常都要追问根因"**
- **从"上线后出了问题再说"到"发版前主动检查 Checklist"**
- **从"这个异常是偶发的不用管"到"偶发异常也要有根因分析"**
- 稳定性成为了每个开发在评估自己代码质量时的默认维度，而不仅仅是 QA 或 SRE 的事情

## 七、可复用的方法论总结

如果将这套稳定性治理体系的实践经验抽象为通用方法论，我认为最核心的几条原则是：

### 原则一：体系大于单点

一个 Crash 被修复了，只说明这一次运气好。只有建立了预防、发现、定位、修复、复盘、改进的闭环体系，才能真正提升系统的稳定性。**单点优化解决的是"这次"的问题，体系建设解决的是"这类"的问题。**

### 原则二：可观测性是治理的前提

你无法治理你观测不到的东西。在投入任何稳定性优化之前，先问自己：我能不能知道线上发生了什么？**把问题变得可见，比立刻解决它更重要。**

### 原则三：分层防御，每层设防

理想状态下，一个问题应该在它产生的那一层就被拦截——不需要流到线上才被发现。**灰度层应该截住测试层没发现的问题，Code Review 应该截住开发阶段没有注意到的问题。**

### 原则四：用机制代替依赖

不要依赖个人的责任心和细心，要依赖工具和流程来保证质量。**一个好的机制可以让普通人做出不普通的成果，而差的机制会让优秀的人也犯错。**

### 原则五：核心链路优先

资源永远是有限的。在投入任何稳定性治理工作之前，先定义清楚"哪些场景出了问题是不可接受的"。**核心链路的稳定是底线，非核心场景可以容忍。先守住底线，再谈扩展。**

### 适合面试/答辩的"金句式"表达：

> **"稳定性治理的本质，是通过工程化手段，将系统的不确定性控制在可接受的范围内。"**

> **"不要问'这个 bug 怎么修'，要问'我们的体系为什么没有发现这个 bug'。"**

> **"Crash 率从 2% 降到 0.4% 只是结果，真正的价值是建立了一个让问题不再反复出现的机制。"**

---

## 八、如果重来一次会怎么做

回顾整个过程，有几点如果重来我会做得不一样：

### 8.1 先建可观测性，再谈治理

我们在治理初期投入了大量精力在"修复问题"上，而不是"让问题可见"上。回过头看，这是一个效率不高的路径。**应该先建立完整的可观测性体系，用数据驱动治理优先级，而不是凭经验判断哪个问题更应该被修复。**

### 8.2 尽早引入自动化

Checklist 和 CR 机制在早期靠人工执行，效果不错但效率低。如果能更早引入自动化检查工具（比如在 CI 中自动检查规则、自动比对灰度数据），可以节省大量人力。**AI Code Review 工具 cr-toolkit 是我们在这个方向上的探索，效果远超预期。**

### 8.3 更早地建立事故复盘文化

早期我们倾向于"修完就完了"，没有系统性地沉淀经验。如果能从一开始就建立严格的事故复盘和知识库沉淀机制，很多后续的问题根本不会发生。**复盘不是为了追责，而是为了让整个组织变得更好。**
